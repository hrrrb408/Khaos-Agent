## 1. 架构总览

### 1.1 三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    上层：Python（Agent 逻辑层）               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ Agent 核心│ │ 工具系统  │ │ 记忆系统  │ │ 模式管理器     │ │
│  │ (循环)   │ │ (注册/调度)│ │ (三层)   │ │ (Office/Coding)│ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ 权限引擎  │ │ 子代理   │ │ 上下文   │ │ 模型路由       │ │
│  │ (规则)   │ │ (单层)   │ │ 压缩     │ │ (Provider+Func)│ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────────┘ │
├─────────────────────────────────────────────────────────────┤
│                  中间层：Go（API 网关层）                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ 平台接入  │ │ 认证鉴权  │ │ 限流     │ │ REST/SSE API   │ │
│  │ (CLI/Web)│ │          │ │          │ │ (供前端调用)    │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────────┘ │
├─────────────────────────────────────────────────────────────┤
│                  底层：Rust（性能瓶颈层）                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ Token    │ │ 工具并行  │ │ 沙箱     │ │ FTS5           │ │
│  │ 解析/计数│ │ 执行引擎  │ │ 管理器   │ │ 全文索引       │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 三层职责划分

| 层 | 语言 | 职责 | 不做什么 |
|----|------|------|----------|
| **上层** | Python | Agent 循环、工具定义、记忆、权限判断、模式切换、模型调用 | 不处理并发调度、不做网络接入 |
| **中间层** | Go | 平台接入（CLI/Web/消息平台）、认证、限流、API 路由 | 不参与 Agent 决策逻辑 |
| **底层** | Rust | Token 解析计数、工具并行执行、Docker 沙箱管理、FTS5 索引 | 不做业务逻辑 |

### 1.3 为什么这样分层

| 决策 | 理由 |
|------|------|
| Python 在上层 | AI/ML 生态最丰富，prompt 工程、工具 JSON Schema、记忆逻辑都是 I/O 密集型，Python 开发效率最高 |
| Go 在中间层 | 高并发、低延迟，天生适合做 API 网关、SSE 长连接、多平台接入 |
| Rust 在底层 | Token 解析和计数是 CPU 密集型高频操作，工具并行执行需要零开销抽象，Docker 管理需要内存安全 |

### 2.6 子代理系统

```
┌─────────────────────────────────────────────┐
│          主 Agent                            │
│                                              │
│    任务 A ──→ spawn ──→ 子代理 A（独立上下文）│
│    任务 B ──→ spawn ──→ 子代理 B（独立上下文）│
│    任务 C ──→ spawn ──→ 子代理 C（独立上下文）│
│                                              │
│  约束:                                        │
│    - 单层，不嵌套                              │
│    - 最大并发数可配置（默认 3）                │
│    - 子代理完成后结果返回主 Agent              │
│    - 子代理拥有独立终端会话                    │
│    - 子代理不持有用户交互能力（不能 ask user） │
└─────────────────────────────────────────────┘
```

### 2.7 上下文压缩

```
┌─────────────────────────────────────────────┐
│            上下文压缩管线                      │
│                                              │
│  Level 0: Micro-compact                      │
│    单条消息过长 → 截断，零 API 开销           │
│                                              │
│  Level 1: Context collapse                   │
│    历史消息累积 → 折叠为摘要                  │
│                                              │
│  Level 2: Auto-compact                       │
│    超 token 阈值 → 模型压缩对话              │
│                                              │
│  Level 3: Session memory                     │
│    关键信息 → 持久化到记忆系统                │
│                                              │
│  触发条件:                                    │
│    token_count > threshold → Level 2         │
│    连续 3 次压缩失败 → 熔断停止重试           │
└─────────────────────────────────────────────┘
```

### 3.2 与 Python 上层通信

```
Go 网关 ←→ Python Agent 层

通信方式: gRPC（内部进程通信）

消息流:
  1. 用户消息 → Go 网关 → gRPC → Python Agent
  2. Python Agent 流式输出 → gRPC stream → Go 网关 → SSE → 前端
  3. 权限确认请求 → Python → Go → SSE permission_request 事件 → 前端 → 用户确认 → HTTP POST /api/chat/:id/confirm → 返回
```

## 6. 交互流程

### 6.1 用户发送消息完整流程

```
用户输入 "帮我写个排序算法"
    │
    v
[Go 网关] 接收请求，认证 + 限流
    │
    v
[Go 网关] gRPC 转发给 Python Agent 层
    │
    v
[Python] 模式管理器判断 → 当前 Office 模式
    │         检测到 coding 任务 → 提醒用户是否切换
    │         用户确认 → 切换到 Coding 模式
    v
[Python] Agent 核心循环启动
    │
    v
[Python] 组装上下文
    │  - system prompt（Coding 模式）
    │  - 记忆注入（Coding 记忆 + 全局记忆）
    │  - KHAOS.md / AGENTS.md 等上下文文件
    │  - 对话历史
    v
[Python] 模型路由 → 选 Coding 模型
    │
    v
[Python] 调用模型 API（SSE 流式）
    │
    v
[模型] 返回 tool_use: write_file("sort.py", ...)
    │
    v
[Python] 权限检查 → ask-every → 请求用户确认
    │
    v
[Go 网关] SSE permission_request 事件 → 前端 → 用户点击确认 → HTTP POST /api/chat/:id/confirm
    │
    v
[Python] 收到确认 → Rust 工具执行器写入文件
    │
    v
[Python] 工具结果追加到消息历史
    │
    v
[Python] 上下文压缩检查 → 未超阈值
    │
    v
[Python] 记忆更新 → 检测到 coding 偏好 → 写入 Coding 记忆
    │
    v
[Python] continue → 回到循环顶部 → 模型继续输出
    │
    v
[模型] 返回 end_turn → 循环结束
    │
    v
[Python] 最终响应 → Go 网关 → SSE → 前端渲染
```

## 9. 错误处理策略

| 错误类型 | 检测层 | 恢复策略 |
|----------|--------|----------|
| 模型 API 超时 | Python | fallback 到备用模型 |
| 模型 API 限流 (429) | Python | 指数退避重试（max 3 次） |
| 工具执行失败 | Rust | 错误信息返回 Python → 模型重决策 |
| prompt-too-long | Python | 触发 auto-compact |
| 流式中途失败 | Python | 清理孤儿消息，重建状态 |
| 压缩连续失败 | Python | 熔断（3 次后停止重试，直接截断） |
| Docker 容器异常 | Rust | 销毁容器，重新创建 |
| gRPC 通信中断 | Go | 自动重连 + 消息队列缓冲 |
| SQLite 锁冲突 | Python | 重试 + WAL 模式 |
