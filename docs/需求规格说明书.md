## 1. 引言

### 1.1 目的

本文档定义 **Khaos（混沌）** 的功能需求和非功能需求，作为后续概要设计、详细设计和开发的依据。

### 1.2 范围

Khaos 是一个通用 AI Agent 平台，具备双模式切换能力：
- **办公模式（Office Mode）**：通用 agent，多工具、多模型、对话式交互
- **Coding 模式（Coding Mode）**：agentic coding，对标 Claude Code + Codex

以 Hermes Agent 为设计参考，从零打造，融合六大竞品的核心设计优势。

### 1.3 术语

| 术语 | 说明 |
|------|------|
| Khaos | 产品名称，英文代号。中文名"混沌" |
| 办公模式 | 通用 agent 模式，对话式交互，多确认 |
| Coding 模式 | 代码开发专用模式，少问多做，自主循环 |
| 双模式切换 | 运行时切换办公/Coding模式，本质是换 system prompt + 工具集 + 模型偏好 + 交互风格 |
| 子代理 | Agent 内部 spawned 的独立 agent 实例，用于并行任务 |
| Docker 沙箱 | Coding 模式下代码执行的隔离环境 |

## 3. 功能需求

### 3.1 核心架构

#### FR-001：Agent 核心循环

**优先级**：P0 | **模式**：通用

Agent 核心循环采用 `while(true)` 架构（参考 Claude Code query.ts）：

1. 构建 system prompt（含上下文文件、模式配置、用户偏好）
2. 调用模型 API（流式响应）
3. 判断停止条件：stop_reason != tool_use → 退出
4. stop_reason == tool_use → 执行工具
5. 工具结果追加到消息历史
6. 上下文压缩检查（超阈值时自动压缩）
7. 回到循环顶部

**关键设计**（学 Claude Code）：
- 流式 API + prompt cache
- 工具结果流式返回（模型思考的同时工具已开始运行）
- maxTurns / maxBudget / abort 多重退出条件

#### FR-002：双模式切换

**优先级**：P0 | **模式**：通用

运行时切换办公模式和 Coding 模式。切换本质：

| 维度 | 办公模式 | Coding 模式 |
|------|----------|-------------|
| System Prompt | 通用 agent 指令 | coding 专精指令 |
| 工具集 | 全量（记忆/技能/cron/浏览器/桌面控制） | coding 子集（文件/终端/git/沙箱） |
| 模型偏好 | 通用模型优先 | 最强 coding 模型优先 |
| 交互风格 | 对话式，多确认 | 少问多做，自主循环 |
| 子代理策略 | 平级委托 | 层级调度 |

切换触发方式：命令切换为主（`/mode coding` / `/mode office`）+ 检测到疑似任务时主动提醒用户确认

#### FR-003：上下文压缩

**优先级**：P0 | **模式**：通用

三层压缩策略（参考 Claude Code + Hermes 扩展为五层）：

| 层级 | 触发条件 | 方式 |
|------|----------|------|
| Micro-compact | 单条消息过长 | 截断，零 API 开销 |
| Context collapse | 历史消息累积 | 折叠为摘要 |
| Auto-compact | 超 token 阈值 | 模型压缩对话 |
| Session memory | 跨压缩保持 | 关键信息持久化（参考 Claude Code sessionMemory） |
| 梦境整合 | 空闲时 | 后台整合记忆碎片（参考 Claude Code autoDream） |

#### FR-004：多模型路由

**优先级**：P1 | **模式**：办公模式优先

支持多模型提供商接入，按任务类型自动选模型：

- Fallback 链（模型不可用时自动切换，参考 OpenClaw）
- 双层路由配置：Provider层（多提供商）+ Function层（每个功能可绑定不同模型，代码不写死）
- 当前接入：NVIDIA NIM
- 后续接入：OpenAI、Anthropic、本地 GGUF 等

### 3.2 工具系统

#### FR-005：工具注册与调度

**优先级**：P0 | **模式**：通用

工具通过声明式注册（JSON Schema），运行时按需加载：

- 每个工具定义：名称、描述、参数 schema、权限等级
- 工具策略管道（参考 OpenClaw 7 层过滤）
- 只读工具并行执行，写操作串行（参考 Claude Code）
- 工具执行超时机制

#### FR-006：文件操作工具

**优先级**：P0 | **模式**：双模式

| 工具 | 说明 |
|------|------|
| read_file | 读取文件（分页、行号） |
| write_file | 写入文件（全量覆盖） |
| patch | 原子 find-and-replace（模糊匹配） |
| search_files | 内容搜索（ripgrep）和文件搜索（glob） |

参考 Hermes 现有实现。

#### FR-007：终端工具

**优先级**：P0 | **模式**：双模式

| 工具 | 说明 |
|------|------|
| terminal | 执行 shell 命令（前台/后台） |
| process | 管理后台进程（poll/kill/log/wait） |

关键设计（参考 Codex）：
- 命令分段安全评估（拆解管道、重定向）
- 权限分级：只读命令自动通过，写/删除需确认
- 超时控制

#### FR-008：Docker 沙箱工具

**优先级**：P0 | **模式**：Coding 模式

| 工具 | 说明 |
|------|------|
| sandbox_exec | 在 Docker 容器中执行命令 |
| sandbox_build | 构建 Docker 镜像 |

关键设计（参考 Codex）：
- 容器生命周期管理（创建/启动/停止/销毁）
- 网络隔离（可选联网）
- 文件系统映射（项目目录挂载）
- 资源限制（CPU/内存/超时）

#### FR-009：Git 工具

**优先级**：P1 | **模式**：Coding 模式优先

| 工具 | 说明 |
|------|------|
| git_diff | 查看 diff |
| git_commit | 提交变更 |
| git_branch | 分支管理 |
| git_log | 查看历史 |

参考 Claude Code 的 git status 自动抓取。

#### FR-010：浏览器工具

**优先级**：P1 | **模式**：办公模式优先

| 工具 | 说明 |
|------|------|
| browser_navigate | 打开网页 |
| browser_click | 点击元素 |
| browser_type | 输入文本 |
| browser_snapshot | 获取页面快照 |
| browser_vision | 页面截图 |

参考 Hermes 现有实现。

#### FR-011：代码搜索与分析工具

**优先级**：P1 | **模式**：Coding 模式

参考 Claude Code 的代码库理解能力：
- 启动时自动抓取 git status、分支、最近提交
- 扫描项目配置文件（如 CLAUDE.md / KHAOS.md）
- AST 级别的代码理解（**Phase 2+**）

### 3.3 子代理系统

#### FR-012：子代理调度

**优先级**：P1 | **模式**：通用

支持 spawned 子代理执行并行/独立任务：

| 维度 | 说明 |
|------|------|
| 隔离性 | 子代理拥有独立上下文，不污染父代理 |
| 并行 | 支持同时运行多个子代理（最大并发可配置，默认3） |
| 嵌套 | 不嵌套，单层。子代理是叶子节点，不再 spawn 子代理 |
| 汇总 | 子代理完成后结果返回父代理 |

参考：Hermes delegate_task + Marvis 四级调度 + Claude Code 三层协作。

---

### 3.4 权限系统

#### FR-013：权限分级与审批

**优先级**：P0 | **模式**：通用

多维度权限设计（参考 Codex 三维度交叉）：

| 维度 | 级别 |
|------|------|
| **审批模式** | auto-approve / suggest / ask-every / deny |
| **沙箱要求** | 无沙箱 / 本地沙箱 / Docker 沙箱 |
| **协作范围** | 只读 / 项目内 / 全系统 |

实现：
- 规则引擎（allow/deny/ask + glob 匹配，参考 Claude Code）
- 权限持久化（用户选择记住后不再重复询问）
- 环境变量清洗（防 API key 泄露）

---

### 3.5 记忆系统

#### FR-014：三层记忆架构

**优先级**：P1 | **模式**：通用

> 详细设计参见《对比分析报告》第十四章。

| 层 | 说明 | 加载方式 |
|----|------|----------|
| 全局记忆 | 两模式共享（用户身份、偏好、环境信息） | 始终加载 |
| 办公记忆 | 仅办公模式（对话历史、任务状态） | 办公模式加载 |
| Coding 记忆 | 仅 Coding 模式（项目上下文、代码习惯） | Coding 模式加载 |
| 意图暂存区 | 跨模式桥梁（切换时保持当前任务上下文） | 切换时传递 |

子功能：
- 记忆衰减机制（TTL + 访问频率）
- 记忆冲突处理（置信度 + 优先级）
- Agent 主动记忆（观察行为自动记录）
- Token 预算管理（L0/L1/L2 分级注入）

### 3.6 交互界面

#### FR-015：CLI 界面

**优先级**：P0 | **模式**：通用

- 基于 TUI（Terminal UI）框架（参考 Hermes Ink/React）
- 支持斜杠命令（`/mode`、`/tools`、`/memory` 等）
- 流式输出（打字机效果）
- 支持 Markdown 渲染（表格、代码块）
- 模式切换指示器（当前模式 + 当前模型）

#### FR-016：Web 界面

**优先级**：P1 | **模式**：通用

- 基于 Web 的对话界面
- 与 CLI 共享 agent 核心
- 技术栈：Next.js（React），后端 API 由 Go 层提供

---

### 3.7 错误处理与恢复

#### FR-017：错误分类与恢复

**优先级**：P0 | **模式**：通用

参考 Claude Code 五种恢复策略 + Hermes error_classifier：

| 错误类型 | 恢复策略 |
|----------|----------|
| 模型超时 | fallback 到备用模型 |
| API 限流 | 指数退避重试 |
| 工具执行失败 | 错误信息返回模型，自动重试/换方案 |
| prompt-too-long | 触发 auto-compact |
| 流式中途失败 | 清理孤儿消息，重建执行状态 |
| 压缩失败 | 熔断机制（连续 3 次失败停止重试） |

## 4. 非功能需求

### 4.1 性能

| 指标 | 要求 |
|------|------|
| 首次响应延迟 | < 2s（首 token） |
| 工具执行延迟 | 取决于工具本身，框架开销 < 100ms |
| 上下文压缩耗时 | < 5s（模型压缩） |
| 并发子代理 | ≥ 3 个同时运行 |

### 4.2 安全

| 要求 | 说明 |
|------|------|
| API Key 保护 | 环境变量存储，日志脱敏 |
| 沙箱隔离 | Coding 模式命令默认在 Docker 中执行 |
| 权限最小化 | 默认 ask-every 模式，用户逐步授权 |
| 文件系统保护 | 规则引擎限制可访问路径 |

### 4.3 可维护性

| 要求 | 说明 |
|------|------|
| 模块化 | 工具、模型、记忆、子代理各自独立模块 |
| 配置化 | 模型、权限、工具集通过配置文件管理 |
| 日志 | 结构化日志（JSON），支持审计 |
| 测试 | Python: pytest / Go: go test / Rust: cargo test（各层用各语言标配），覆盖率目标 80%+ |

### 4.4 可扩展性

| 要求 | 说明 |
|------|------|
| 工具插件化 | 声明式注册，新增工具无需改核心代码 |
| 模型可插拔 | 新增模型提供商只需实现 adapter |
| 模式可扩展 | 框架支持新增模式（未来可加本地模式、专家模式等） |
