### 1.1 Agent 核心

#### 1.1.1 core.py — Agent 主循环

每次用户输入先由 `TurnCoordinator` 创建 durable TurnId/AttemptId。`agent_turn_events`
使用 per-turn 单调 sequence；tool.call、approval.wait、tool.result 和 terminal 均通过单写者
CAS 追加。terminal event 与 `agent_turns.status` 同事务提交，对外 done/error 只能在该事务
成功后发送。启动首次 turn 前把旧 running turn 标记 interrupted，不根据消息猜测成功。

```python
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional
import asyncio


class StopReason(Enum):
    END_TURN = "end_turn"
    TOOL_USE = "tool_use"
    MAX_TURNS = "max_turns"
    MAX_BUDGET = "max_budget"
    USER_ABORT = "user_abort"
    ERROR = "error"


@dataclass
class AgentConfig:
    max_turns: int = 100          # 单轮对话最大工具调用次数
    max_budget_tokens: int = 500000  # 单轮对话最大 token 消耗
    stream_timeout: int = 120     # 流式响应超时（秒）
    compression_threshold: int = 128000  # 触发 auto-compact 的 token 阈值


@dataclass
class Message:
    role: str                    # "system" | "user" | "assistant" | "tool"
    content: str
    tool_calls: list[dict] = field(default_factory=list)
    tool_call_id: Optional[str] = None
    token_count: int = 0
    created_at: float = 0.0


class AgentLoop:
    """Agent 核心循环 (while true)"""

    def __init__(
        self,
        config: AgentConfig,
        mode_manager: "ModeManager",
        tool_registry: "ToolRegistry",
        permission_engine: "PermissionEngine",
        memory_manager: "MemoryManager",
        context_compressor: "ContextCompressor",
        router: "ModelRouter",
        token_engine,            # Rust FFI
        tool_executor,            # Rust FFI
        db: "Database",
    ):
        self.config = config
        self.mode_manager = mode_manager
        self.tool_registry = tool_registry
        self.permission_engine = permission_engine
        self.memory_manager = memory_manager
        self.compressor = context_compressor
        self.router = router
        self.token_engine = token_engine
        self.tool_executor = tool_executor
        self.db = db

    async def run(self, user_input: str, session_id: str) -> AsyncIterator[Message]:
        """
        主循环入口。返回流式 Message 迭代器。

        状态机: IDLE → BUILDING_CONTEXT → CALLING_MODEL → EXECUTING_TOOLS → COMPRESSING → DONE/ERROR

        消息落库时机: 每条消息（用户/助手/工具）立即写入 messages 表

        SSE 事件生成规则:
          - 模型返回 text chunk → SSE text 事件
          - 模型返回 tool_use → SSE tool_call 事件 → 检查权限 → SSE permission_request（如果需要确认）→ 执行工具 → SSE tool_result 事件
          - 循环直到 stop_reason != tool_use

        工具调用失败处理: failure 不算致命错误，ToolResult(success=False, error=...) 返回模型重决策，不抛异常

        上下文压缩检查点: 每次循环迭代前检查 total_tokens > threshold
        超预算退出: total_tokens > max_budget_tokens → 生成 done 事件
        """
        # === 状态机初始化 ===
        state = "IDLE"
        turn_count = 0
        total_tokens = 0

        try:
            # === IDLE → BUILDING_CONTEXT ===
            state = "BUILDING_CONTEXT"
            messages = await self._build_context(session_id)

            # 用户消息落库
            user_msg = Message(role="user", content=user_input)
            messages.append(user_msg)
            await self.db.insert_message(session_id, user_msg)  # 立即落库

            # === 主循环 ===
            while turn_count < self.config.max_turns:
                # 上下文压缩检查点：每次迭代前检查
                if await self._check_compression(messages):
                    state = "COMPRESSING"
                    result = await self.compressor.compress(
                        messages, self.config.compression_threshold
                    )
                    messages = result.messages
                    total_tokens = sum(m.token_count for m in messages)
                    # 压缩摘要消息落库
                    for msg in messages:
                        if msg.token_count > 0:
                            await self.db.update_message_tokens(msg)
                    state = "CALLING_MODEL"

                # 超预算退出
                if total_tokens > self.config.max_budget_tokens:
                    yield Message(
                        role="assistant",
                        content="",
                        token_count=total_tokens,
                    )
                    yield Message(
                        role="system",
                        content=f"demand: max_budget exceeded ({total_tokens} > {self.config.max_budget_tokens})",
                    )
                    return

                # === CALLING_MODEL ===
                state = "CALLING_MODEL"
                assistant_content = ""
                tool_calls = []
                stop_reason = None

                async for chunk in self.router.call("agent_loop", messages):
                    # text chunk → SSE text 事件
                    if chunk.content:
                        assistant_content += chunk.content
                        chunk.token_count = self.token_engine.count_tokens(chunk.content)
                        total_tokens += chunk.token_count
                        yield chunk  # SSE text 事件

                    if chunk.tool_calls:
                        tool_calls.extend(chunk.tool_calls)

                    if hasattr(chunk, "stop_reason") and chunk.stop_reason:
                        stop_reason = chunk.stop_reason

                turn_count += 1

                # 助手消息落库（包含 tool_calls）
                assistant_msg = Message(
                    role="assistant",
                    content=assistant_content,
                    tool_calls=tool_calls,
                    token_count=self.token_engine.count_tokens(assistant_content),
                )
                messages.append(assistant_msg)
                await self.db.insert_message(session_id, assistant_msg)  # 立即落库

                # === 判断 stop_reason ===
                if stop_reason != "tool_use":
                    # end_turn / max_turns / error → 退出
                    state = "DONE"
                    return

                # === EXECUTING_TOOLS ===
                state = "EXECUTING_TOOLS"
                for tool_result_msg in await self._execute_tool_calls(tool_calls):
                    # tool_result → SSE tool_result 事件
                    yield tool_result_msg
                    messages.append(tool_result_msg)
                    total_tokens += tool_result_msg.token_count
                    await self.db.insert_message(session_id, tool_result_msg)  # 立即落库

            # max_turns 退出
            state = "DONE"

        except asyncio.CancelledError:
            state = "ERROR"
            raise

        except Exception as e:
            state = "ERROR"
            logger.error("Agent loop error: %s", e, exc_info=True)
            yield Message(role="system", content=f"error: {e}")


    async def _build_context(self, session_id: str) -> list[Message]:
        """组装上下文: system prompt + 记忆注入 + 上下文文件 + 历史消息"""
        ...

    async def _call_model(
        self, messages: list[Message]
    ) -> AsyncIterator[Message]:
        """调用模型 API，流式返回 Message chunks"""
        ...

    async def _execute_tool_calls(
        self, tool_calls: list[dict]
    ) -> list[Message]:
        """
        批量执行工具调用。
        只读工具并行 (Rust executor)，写操作串行。
        每个工具调用前经过权限检查。
        """
        ...

    async def _check_compression(self, messages: list[Message]) -> bool:
        """检查是否需要触发上下文压缩"""
        total_tokens = sum(m.token_count for m in messages)
        return total_tokens > self.config.compression_threshold
```

#### 1.1.3 compressor.py — 上下文压缩

```python
from enum import Enum
from dataclasses import dataclass


class CompressionLevel(Enum):
    MICRO_COMPACT = 0      # 单条消息截断
    CONTEXT_COLLAPSE = 1   # 折叠为摘要
    AUTO_COMPACT = 2       # 模型压缩
    SESSION_MEMORY = 3     # 持久化到记忆


@dataclass
class CompressionResult:
    level: CompressionLevel
    original_tokens: int
    compressed_tokens: int
    messages: list[Message]  # 压缩后的消息列表


class ContextCompressor:
    """三层上下文压缩 + 熔断机制"""

    def __init__(self, router: "ModelRouter", token_engine, memory_manager: "MemoryManager"):
        self.router = router
        self.token_engine = token_engine
        self.memory_manager = memory_manager
        self._consecutive_failures = 0

    async def compress(
        self, messages: list[Message], threshold: int
    ) -> CompressionResult:
        """
        按阈值自动选择压缩层级:
        - token > threshold * 1.5 → Level 2 (模型压缩)
        - token > threshold * 1.2 → Level 1 (摘要折叠)
        - 单条消息 > 10000 chars → Level 0 (截断)

        压缩边界保护：
          - tool_call 和对应的 tool_result 必须成对保留，不能只压缩其中一个
          - 最近 4 条消息（2轮对话）始终保留完整
          - system prompt 不参与压缩

        摘要格式：[摘要开始] 原始 N 条消息压缩为 1 条。包含关键决策: ... [摘要结束]

        失败回退：
          - Level 2 模型压缩失败 → 降级到 Level 1 context-collapse
          - Level 1 失败 → 降级到 Level 0 micro-compact（粗暴截断最早的消息）
          - 连续 3 次 Level 2 失败 → 熔断器打开，后续只用 Level 0/1
        """
        original_tokens = sum(self.token_engine.count_tokens(m.content) for m in messages)

        # === 压缩边界保护 ===
        # 分离不可压缩的消息
        system_msgs = [m for m in messages if m.role == "system"]
        recent_msgs = messages[-4:] if len(messages) > 4 else []  # 最近 4 条（2轮）
        compressible_start = len(system_msgs)
        compressible_end = len(messages) - len(recent_msgs)
        compressible = messages[compressible_start:compressible_end]

        if len(compressible) <= 1:
            return CompressionResult(
                level=CompressionLevel.MICRO_COMPACT,
                original_tokens=original_tokens,
                compressed_tokens=original_tokens,
                messages=messages,
            )

        # === Level 0: 单条消息截断（零 API 开销） ===
        for msg in compressible:
            if len(msg.content) > 10000:
                msg = await self._micro_compact(msg)
                messages[compressible_start] = msg

        total_tokens = sum(self.token_engine.count_tokens(m.content) for m in messages)
        if total_tokens <= threshold:
            return CompressionResult(
                level=CompressionLevel.MICRO_COMPACT,
                original_tokens=original_tokens,
                compressed_tokens=total_tokens,
                messages=messages,
            )

        # === 确定压缩层级 ===
        if total_tokens > threshold * 1.5 and not self.is_circuit_open:
            # Level 2: 模型压缩
            try:
                collapsed = await self._auto_compact(compressible)
                # 合并：system + collapsed + recent
                messages = system_msgs + collapsed + recent_msgs
                self._consecutive_failures = 0  # 成功，重置计数
                compressed_tokens = sum(
                    self.token_engine.count_tokens(m.content) for m in messages
                )
                return CompressionResult(
                    level=CompressionLevel.AUTO_COMPACT,
                    original_tokens=original_tokens,
                    compressed_tokens=compressed_tokens,
                    messages=messages,
                )
            except Exception as e:
                # Level 2 失败 → 降级到 Level 1
                logger.warning("Level 2 compression failed: %s, falling back to Level 1", e)
                self._consecutive_failures += 1
                if self._consecutive_failures >= 3:
                    logger.error("Circuit breaker OPEN: 3 consecutive Level 2 failures")

        # === Level 1: 折叠为摘要 ===
        try:
            collapsed = await self._context_collapse(compressible)
            messages = system_msgs + collapsed + recent_msgs
            compressed_tokens = sum(
                self.token_engine.count_tokens(m.content) for m in messages
            )
            return CompressionResult(
                level=CompressionLevel.CONTEXT_COLLAPSE,
                original_tokens=original_tokens,
                compressed_tokens=compressed_tokens,
                messages=messages,
            )
        except Exception as e:
            # Level 1 失败 → 降级到 Level 0（粗暴截断）
            logger.warning("Level 1 compression failed: %s, falling back to Level 0", e)
            # Level 0: 截断最早的 compressible 消息
            truncated = compressible[:1]
            for msg in truncated:
                msg = await self._micro_compact(msg, max_chars=2000)
            messages = system_msgs + [truncated[0]] + recent_msgs
            compressed_tokens = sum(
                self.token_engine.count_tokens(m.content) for m in messages
            )
            return CompressionResult(
                level=CompressionLevel.MICRO_COMPACT,
                original_tokens=original_tokens,
                compressed_tokens=compressed_tokens,
                messages=messages,
            )

    async def _micro_compact(self, message: Message, max_chars: int = 10000) -> Message:
        """
        Level 0: 截断单条过长消息，零 API 开销。
        在 max_chars 处截断，附加截断标记。
        """
        if len(message.content) <= max_chars:
            return message
        truncated = message.content[:max_chars]
        return Message(
            role=message.role,
            content=f"{truncated}\n\n[截断: 原文 {len(message.content)} 字符，已保留前 {max_chars}]",
            tool_calls=message.tool_calls,
            tool_call_id=message.tool_call_id,
        )

    async def _context_collapse(self, messages: list[Message]) -> list[Message]:
        """
        Level 1: 折叠历史消息为摘要（保留最近 N 条完整）。
        将可压缩消息折叠为一条摘要消息。
        确保工具调用/工具结果成对保留。
        """
        # 收集成对的 tool_call + tool_result，确保不拆散
        protected_pairs = []
        unprotected = []
        i = 0
        while i < len(messages):
            msg = messages[i]
            if msg.tool_calls:
                # 查找对应的 tool result
                tool_call_ids = [tc.get("id") for tc in msg.tool_calls]
                tool_results = []
                j = i + 1
                while j < len(messages) and len(tool_results) < len(tool_call_ids):
                    if messages[j].role == "tool" and messages[j].tool_call_id in tool_call_ids:
                        tool_results.append(messages[j])
                        j += 1
                    else:
                        break
                # 成对保留
                protected_pairs.append(msg)
                protected_pairs.extend(tool_results)
                i = j
            else:
                unprotected.append(msg)
                i += 1

        # 将 unprotected 消息折叠为摘要
        if not unprotected:
            return messages

        summary_content = (
            f"[摘要开始] 原始 {len(unprotected)} 条消息压缩为 1 条。"
            f"包含关键决策: {extract_key_decisions(unprotected)} [摘要结束]"
        )
        summary_msg = Message(role="assistant", content=summary_content)

        return protected_pairs + [summary_msg]

    async def _auto_compact(self, messages: list[Message]) -> list[Message]:
        """
        Level 2: 调用模型压缩对话。连续 3 次失败触发熔断。
        将消息列表发给压缩模型，生成摘要。
        确保工具调用/工具结果成对保留。
        """
        # 构建压缩请求
        compress_prompt = (
            "请将以下对话历史压缩为简洁的摘要，保留关键决策、"
            "工具调用及其结果、重要的事实和数字。"
            "格式要求：[摘要开始] 原始 N 条消息压缩为 1 条。"
            "包含关键决策: ... [摘要结束]\n\n"
            + "\n".join(
                f"[{m.role}]: {m.content[:500]}{'...' if len(m.content) > 500 else ''}"
                for m in messages
            )
        )

        # 调用压缩模型
        response = await self.router.call("compression", [Message(role="user", content=compress_prompt)])
        summary = "".join(chunk.content async for chunk in response)

        return [Message(role="assistant", content=summary)]

    @property
    def is_circuit_open(self) -> bool:
        """熔断器是否打开"""
        return self._consecutive_failures >= 3

    def _reset_circuit(self):
        self._consecutive_failures = 0
```

#### 1.3.2 scheduler.py — 工具调度

```python
from dataclasses import dataclass
from typing import Any


@dataclass
class ToolResult:
    tool_call_id: str
    name: str
    success: bool
    output: Any
    error: str = ""
    duration_ms: int = 0
    arguments: dict[str, Any] | None = None  # 原始调用参数，供任务追踪使用


@dataclass
class ToolBudget:
    max_calls: int = 50              # 最大工具调用次数
    max_output_chars: int = 100000   # 最大工具输出总字符数
    _call_count: int = 0
    _output_chars: int = 0

    @property
    def is_exhausted(self) -> bool:
        return self._call_count >= self.max_calls or self._output_chars >= self.max_output_chars

    def record(self, output_chars: int):
        self._call_count += 1
        self._output_chars += output_chars


class ToolScheduler:
    """工具调度器 — 并行/串行分发"""

    def __init__(
        self,
        registry: ToolRegistry,
        permission_engine: "PermissionEngine",
        tool_executor,          # Rust FFI
        budget: ToolBudget,
    ):
        self.registry = registry
        self.permission_engine = permission_engine
        self.tool_executor = tool_executor
        self.budget = budget

    async def execute_batch(
        self, tool_calls: list[dict], mode: str
    ) -> list[ToolResult]:
        """
        批量执行工具调用。

        具体流程：
          1. 预算检查：budget.is_exhausted → 返回空结果 + 警告
          2. 遍历 tool_calls，逐个调用 permission_engine.check()
             - 返回 "deny" → 跳过，返回 ToolResult(success=False, error="denied")
             - 返回 "ask" → 暂停批量，发送 permission_request SSE 事件，等用户确认后继续
          3. 权限全部通过后，调用 registry.get_parallel_tools() 分组
          4. 并行组 → rust_tool_executor.parallel(calls, timeout)
          5. 串行组 → 依次调用，每个结果检查 success，记录预算
          6. 如果 budget 耗尽 → 中断剩余工具调用，返回已完成的 + 警告

        失败策略：单个工具失败不中断其他工具
        """
        results: list[ToolResult] = []

        # === 1. 预算检查 ===
        if self.budget.is_exhausted:
            logger.warning("Tool budget exhausted, skipping all %d tool calls", len(tool_calls))
            return results

        # === 2. 权限检查（逐个） ===
        approved_calls: list[dict] = []
        for call in tool_calls:
            tool_name = call["name"]
            params = call.get("arguments", {})
            tool_def = self.registry.get(tool_name)

            decision = await self.permission_engine.check(
                tool_name=tool_name,
                params=params,
                permission_level=tool_def.permission_level,
                mode=mode,
            )

            if decision.approved.value == "deny":
                # 拒绝 → 跳过，返回 denied 结果
                results.append(ToolResult(
                    tool_call_id=call["id"],
                    name=tool_name,
                    success=False,
                    error=f"Permission denied: {decision.reason}",
                ))
            elif decision.requires_user_confirm:
                # ask → 暂停批量，发送 permission_request SSE 事件
                # 等 HTTP POST /api/chat/:id/confirm 后继续
                yield_sse_event("permission_request", {
                    "id": call["id"],
                    "name": tool_name,
                    "arguments": params,
                    "level": tool_def.permission_level,
                })
                confirmed = await wait_for_user_confirm(call["id"])
                if not confirmed:
                    results.append(ToolResult(
                        tool_call_id=call["id"],
                        name=tool_name,
                        success=False,
                        error="User denied permission",
                    ))
                    continue
                # 用户确认，如果带 remember=True → 生成并持久化规则
                if confirmed.get("remember"):
                    await self.permission_engine.grant_rule(
                        PermissionRule(
                            pattern=normalize_target(tool_name, params),
                            permission_level=tool_def.permission_level,
                            approval=decision.approved,
                            mode=mode,
                        )
                    )
                approved_calls.append(call)
            else:
                # allow → 直接加入
                approved_calls.append(call)

        if not approved_calls:
            return results

        # === 3. 分组: 并行组 + 串行组 ===
        parallel_calls, serial_calls = self.registry.get_parallel_tools(approved_calls)

        # === 4. 并行组 → Rust executor 并发执行 ===
        if parallel_calls:
            try:
                parallel_results = self.tool_executor.execute_tools_parallel(
                    parallel_calls,
                    timeout_ms=get_global_timeout(),
                )
                for i, pr in enumerate(parallel_results):
                    tool_result = ToolResult(
                        tool_call_id=parallel_calls[i]["id"],
                        name=parallel_calls[i]["name"],
                        success=pr.success,
                        output=pr.output,
                        error=pr.error or "",
                    )
                    results.append(tool_result)
                    self.budget.record(len(str(pr.output)))
            except Exception as e:
                # 整组失败 → 为每个调用生成失败结果
                for call in parallel_calls:
                    results.append(ToolResult(
                        tool_call_id=call["id"],
                        name=call["name"],
                        success=False,
                        error=f"Parallel execution failed: {e}",
                    ))

        # === 5. 串行组 → 依次执行 ===
        for call in serial_calls:
            if self.budget.is_exhausted:
                # 预算耗尽 → 中断剩余
                logger.warning("Budget exhausted during serial execution, skipping %s", call["name"])
                results.append(ToolResult(
                    tool_call_id=call["id"],
                    name=call["name"],
                    success=False,
                    error="Tool budget exhausted",
                ))
                break

            try:
                tool_def = self.registry.get(call["name"])
                output = await tool_def.handler(**call.get("arguments", {}))
                result = ToolResult(
                    tool_call_id=call["id"],
                    name=call["name"],
                    success=True,
                    output=output,
                )
                self.budget.record(len(str(output)))
                results.append(result)
            except Exception as e:
                # 单个工具失败 → 不中断其他工具
                results.append(ToolResult(
                    tool_call_id=call["id"],
                    name=call["name"],
                    success=False,
                    error=str(e),
                ))

        return results

    async def _check_permission(
        self, tool_name: str, params: dict, mode: str
    ) -> str:
        """检查权限，返回 "allow" | "deny" | "ask" """
        ...
```

#### 1.5.1 engine.py

```python
from enum import Enum
from dataclasses import dataclass


class ApprovalMode(Enum):
    AUTO_APPROVE = "auto-approve"
    SUGGEST = "suggest"
    ASK_EVERY = "ask-every"
    DENY = "deny"


@dataclass
class PermissionRule:
    id: Optional[int]
    pattern: str              # glob 模式 (如 "npm install *")
    permission_level: str     # "read" | "write" | "execute" | "network"
    approval: ApprovalMode     # 该模式下的审批结果
    mode: str                 # "office" | "coding" | "all"
    granted_at: float = 0.0


@dataclass
class PermissionDecision:
    approved: ApprovalMode
    reason: str                # 审批理由（展示给用户）
    matched_rule: Optional[PermissionRule] = None
    requires_user_confirm: bool = False


class PermissionEngine:
    """权限引擎 — 规则匹配 + 用户确认"""

    def __init__(self, db: "Database", default_mode: ApprovalMode = ApprovalMode.ASK_EVERY):
        self.db = db
        self._default_mode = default_mode
        self._rules: list[PermissionRule] = []

    async def load_rules(self) -> None:
        """从数据库加载权限规则"""
        ...

    async def check(
        self,
        tool_name: str,
        params: dict,
        permission_level: str,
        mode: str,
    ) -> PermissionDecision:
        """
        权限检查。

        target 规范化规则：
          - 文件路径：os.path.normpath() + os.path.realpath() 解析后匹配
          - 命令：拆解为 base_command + args，base_command 匹配
          - 网络 URL：解析为 scheme + host 匹配

        glob 匹配：fnmatch.fnmatch()，支持 * ? [seq] [!seq]
        规则遍历顺序：按 granted_at DESC（最新规则优先）
        默认返回：无匹配规则 → PermissionDecision(approved=ApprovalMode.ASK_EVERY, requires_user_confirm=True)

        remember rule 生成：用户确认时如果带 remember=True → 生成 PermissionRule(pattern=规范化后的target, ...) 并持久化
        """
        # === target 规范化 ===
        target = self._normalize_target(tool_name, params)

        # === 遍历规则表（按 granted_at DESC，最新规则优先） ===
        for rule in self._rules:
            # 模式隔离检查：如果规则指定了模式，必须匹配当前模式
            if rule.mode != "all" and rule.mode != mode:
                continue

            # 权限等级检查：规则等级必须匹配或包含
            if rule.permission_level != permission_level:
                continue

            # glob 模式匹配
            if self._match_pattern(rule.pattern, target):
                # 匹配到规则 → 返回对应审批结果
                return PermissionDecision(
                    approved=rule.approval,
                    reason=f"Matched rule: {rule.pattern} (granted at {rule.granted_at})",
                    matched_rule=rule,
                    requires_user_confirm=(rule.approval == ApprovalMode.ASK_EVERY),
                )

        # === 无匹配规则 → 使用默认模式 ===
        if self._default_mode == ApprovalMode.AUTO_APPROVE:
            return PermissionDecision(
                approved=ApprovalMode.AUTO_APPROVE,
                reason="No matching rule, default: auto-approve",
                requires_user_confirm=False,
            )
        elif self._default_mode == ApprovalMode.DENY:
            return PermissionDecision(
                approved=ApprovalMode.DENY,
                reason="No matching rule, default: deny",
                requires_user_confirm=False,
            )
        else:
            # 默认: ask-every
            return PermissionDecision(
                approved=ApprovalMode.ASK_EVERY,
                reason="No matching rule, default: ask-every",
                requires_user_confirm=True,
            )

    async def grant_rule(self, rule: PermissionRule) -> None:
        """用户授权后，持久化权限规则"""
        ...

    async def revoke_rule(self, rule_id: int) -> None:
        """撤销权限规则"""
        ...

    def _match_pattern(self, pattern: str, target: str) -> bool:
        """glob 模式匹配。使用 fnmatch.fnmatch()，支持 * ? [seq] [!seq]"""
        return fnmatch.fnmatch(target, pattern)
```

### 2.3 认证设计

```
所有 Gateway 监听（包括 `127.0.0.1`）都必须具有 API Key。未显式配置
`KHAOS_API_KEY` 时，Gateway 在用户配置目录创建并复用 256-bit 随机 token；token
文件及父目录分别为 `0o600`、`0o700`，symlink、非 regular file、弱 token 或 identity
漂移均 fail closed。空认证不得启动监听，认证 middleware 收到空配置也不得放行。
  - Header: X-Khaos-Key: <api_key>
  - 禁止 URL query 携带凭据；比较使用常量时间算法
  - `/api/health` 可匿名；Chat、Task、Memory、Config、Channel、SubAgent、Audit、Metrics
    等接口必须建立 authenticated principal。平台 Webhook 使用独立签名认证
  - CORS 默认关闭且不返回 `Access-Control-Allow-Origin`；只有
    `KHAOS_CORS_ORIGINS` 中的完整 `http(s)://host[:port]` Origin 才能预检和访问，
    永远不返回 `*`
  - 所有请求先校验 Host allowlist；默认只接受监听地址及 loopback aliases，非回环
    部署通过 `KHAOS_ALLOWED_HOSTS` 显式配置，防止 DNS rebinding
  - 所有 Agent 传输入口进入同一认证、限流和日志中间件；不提供独立 /api/ws/
  - 认证 session/task 在创建时绑定 middleware principal；stream、confirm、mode 与
    task 生命周期端点必须验证 owner，未知 owner fail closed
  - Browser UI 使用带 `X-Khaos-Key` 的 fetch streaming，不使用会泄露 token 的 URL
    query 或无法设置认证 header 的原生 EventSource
  - 所有响应声明 `X-Khaos-Protocol-Version: 1`；Task SSE 使用持久化 sequence 作为
    event id，并接受 `Last-Event-ID` 重放游标
```

### 2.4 限流设计

```
令牌桶算法 (Token Bucket)

对内: 限制 Agent 对模型 API 的调用频率（与 Provider RPM 同步）
对外: 限制用户请求频率（防止误操作刷爆 token）

配置:
  default_rate: 60 req/min
  burst: 10

请求边界:
  普通 JSON 请求最大 1 MiB，平台 webhook 最大 2 MiB；JSON decoder 拒绝未知字段、
  多个顶层值和超限 body。SSE/NDJSON handler 监听 request context，客户端断线时立即
  停止读取上游 channel，使取消沿同一 RPC context 传播。
```

## 5.5 运行时后台服务与平台 Webhook 安全

- `AgentService` 持有进程级 `CronEngine`；JSON-line 服务监听前调用
  `AgentService.start()`，服务退出时在 `finally` 中调用 `shutdown()`。
- JSON-line 控制面仅监听权限为 `0o600` 的 Unix Domain Socket，不允许 TCP
  监听；Go Gateway 通过绝对 UDS 路径和可取消的 `DialContext` 连接。
- Cron 任务通过普通 office-mode Agent 主链路执行，使用 `cron:<task_id>` 作为
  会话 ID；引擎实例注入 `cron_tools`，确保 RPC、Agent 工具共享同一任务状态。
- Discord 入站请求使用 Ed25519 验证 `timestamp + raw_body`，配置的 secret
  必须为 Discord application public key 的十六进制形式。
- 微信 XML 使用 `xml.etree.ElementTree` 解析，并兼容命名空间标签；禁止使用
  正则表达式解释 XML 结构。

## 5.6 剩余风险收敛设计

- 代码理解：Python 使用 AST；Go/Rust 使用零依赖结构化正则提取函数、调用、
  `import`、`mod` 和 `use` 关系。解析失败必须降级为空图，不影响 Agent 主链路。
- 长任务：`coding_tasks` 保存完整 JSON 状态。服务启动恢复任务，并将重启前的
  活跃状态转成 `blocked`，错误注明 `interrupted by process restart`。
- 执行隔离：macOS 终端通过 `sandbox-exec`，Linux 通过 `bwrap`；若父进程已处于
  Codex OS 沙箱则继承父级约束。没有可用 OS 沙箱时拒绝执行。交互式 Python、
  `python -c` 以及非测试用途的 `python -m` 均直接阻断。
- Runtime 按每次请求的 access mode 重新选择执行 backend；read-only 与
  workspace-write 均绑定 active TaskWorkspace。AgentLoop 缺少注入服务时使用
  deny backend，Host backend 不得作为 Agent 生成命令的自动 fallback。
- 每个 ExecutionRequest 在构造时规范化为 schema v1 的不可变
  `PermissionProfile`。filesystem、network、canonical workspace/read-write roots、
  unreadable host-secret roots、environment keys 和 ResourceBudget 只从 profile
  读取；旧字段是兼容投影，显式 profile 与旧字段冲突时以 profile 为准，并把
  profile digest 记录为后续 Approval/Audit binding 输入。
- production ExecutionService 在 active TaskWorkspace 上重新绑定 profile，并验证
  workspace-write 只能获得该 worktree 这一条写根。macOS 使用绝对
  `/usr/bin/sandbox-exec`，从 `deny default` 出发仅允许 TaskWorkspace、命令安装根、
  macOS 系统运行时与合成 HOME/TMP；禁止全局 `allow file-read*`。Linux bwrap 从空
  tmpfs root 出发，只挂载 executable/动态库所需系统目录、最小 `/dev`、隔离 `/proc`、
  有大小上限的 `/tmp`、合成 HOME 与 TaskWorkspace，禁止 `--ro-bind / /`。因此未显式
  挂载的其他仓库、用户文档、浏览器 profile 和 credential store 均不可见。默认环境
  不继承 HOME；工具需要 HOME 时只能使用自己创建的临时目录。请求受限网络后端不
  支持时必须拒绝，不能静默降为禁网或放宽为宿主网络。
- ExecutionService、Host/macOS/Linux adapter、Docker foreground 和 managed stdio
  process 共享一个 `ProcessSupervisor`。注册键为 correlation/execution ID；重复活跃
  ID 拒绝执行。POSIX 子进程必须 `start_new_session`，取消和 timeout 终止整个 process
  group，先 TERM、grace 到期后 KILL；service shutdown 必须等待所有登记进程退出。
- stdout/stderr 使用独立异步 reader 并行 drain，各自最多保留总 output budget 的
  50%。被丢弃字节写入 diagnostics，不写完整临时 artifact；这样同时阻断输出内存 DoS、
  pipe deadlock 和通过 artifact 绕过结果配额。Docker 的容器 stop/kill/rm 只用于资源
  清理，宿主 `docker run` 进程的取消和输出仍由相同 supervisor 管理。
- `ResourceBudget` 的 timeout/output/pids/CPU/memory/tmpfs/file/open-files 不是声明性
  元数据。POSIX child 在 exec 前强制 CPU、单文件大小、fd 数和（Linux）address-space
  rlimit；统一 watchdog 以 50ms 周期统计同 process group 的总 PID/RSS，超限立即终止
  整棵进程树并返回 `resource-exhausted`。Linux `/tmp` 同时由 bwrap `--size` 限制。
- Linux bwrap 的 probe 和 execute 共用 `argv_prefix`，避免“探测配置比生产配置弱”或
  两者漂移。挂载顺序为空 tmpfs root、显式 runtime roots、新 `/dev`、新 `/proc`、
  bounded tmpfs `/tmp`、合成 HOME、最后 bind TaskWorkspace；namespace 至少隔离
  network/PID/IPC/UTS，并启用 new-session 与
  die-with-parent。请求 cwd 映射到 sandbox workdir 下的相对子目录，无法相对化即拒绝。
  probe timeout、启动失败或异常只产生 unavailable，不允许 selector 冒泡后由调用方
  自行选择 Host。Linux CI 必须用 `KHAOS_REQUIRE_PLATFORM_SANDBOX=1` 跑真实隔离矩阵。
- macOS capability probe 必须真实执行与生产相同的 Seatbelt profile，至少证明 worktree
  可写、外部不可写、非 allowlist 文件不可读和 network denial；系统 framework/dylib
  只获得显式 read/map-executable 权限，用户 HOME 不得成为 runtime root。probe 失败或
  抛异常均返回 unsupported。Windows
  尚无强制 backend，因此 selector 返回带平台原因的显式 unsupported，不能走 Host。
- Docker 镜像 allowlist 只允许 sha256 digest，执行固定 `--pull never`。container 使用
  随机 owner nonce label；任何 cleanup 先 inspect label，缺失视为已删除，不匹配则拒绝
  stop/kill/rm。容器采用 read-only rootfs、独立受限 tmpfs、uid/gid 65534、drop ALL、
  no-new-privileges、network none、IPC none、pids/CPU/memory 配额。Docker mount 字符串
  拒绝 delimiter 注入，所有 daemon CLI 也走 ProcessSupervisor。真实 cleanup E2E 只能在
  无用户容器的 CI runner 运行，避免测试本身造成宿主服务中断。
- `sed`、`find`、`awk` 具有写入或启动子命令能力，不属于免确认只读命令集；
  read-only 标签不能替代 OS mount policy。
- 自动化串联：AgentLoop 持久化消息后同步写入 `messages_fts`；工具结果写入任务
  trace；成功任务结束时调用 `SkillGenerator` 并把候选存入任务 metadata。
- Bot 出站：四个平台统一使用注入式 `httpx.AsyncClient`。生产环境执行真实 HTTP，
  测试必须使用 `MockTransport`，禁止访问真实平台。

## 5.7 GPT 审查问题收敛

- CLI、TUI、Gateway 和 SubAgent 必须通过异步 `runtime.build_runtime()` 构造
  `AgentLoop`；Factory 负责加载模式、权限、记忆、技能、安全策略及 Coding 组件。
- 每次 Coding turn 使用独立 `VerifyFixLoop`，任务结束依据 stop reason 写入
  completed、failed 或 cancelled；终止态不得退回活跃态。
- 工具输出在进入 `ToolResult` 前完成 secret 扫描和递归脱敏，私钥块整体替换。
- GitHub 远程操作独立于本地 Git 工具，通过异步 `gh` 子进程实现 PR、Issue、
  评论和 reviewer 工作流。
- Go 网关通过 `/v1/tasks` 暴露创建、查询、审批、拒绝、取消、artifact 与 SSE
  事件流；Python `TaskService` 和持久化 `TaskManager` 是唯一状态源。

## 5.8 GPT 二次审查收敛

- TaskManager 以 `TransitionResult` 报告状态迁移。终止态严格互斥；取消、审批和
  拒绝不会覆盖完成、失败或已取消的任务。
- 每个任务有持久化的单调 `event_sequence`，SSE snapshot 和后续事件使用同一序列。
- Coding task 依次进入 running、blocked、waiting_test、fixing 和终止态；Office
  模式不创建任务也不启用 VerifyFix。
- Runtime Factory 始终创建 task manager、skill generator 和 VerifyFix 工厂，因此
  Office→Coding 切换不需要重建入口；RuntimeResult 提供 `aclose()`，TUI 与 CLI
  分别在退出时释放自身持有的资源。

## 5.9 Principal-bound Tool Approval

- Scheduler 在 permission event 之前注册不可变 `ApprovalBinding`，字段覆盖 principal、
  session、task、turn、tool call、tool name、arguments digest、workspace、permission
  profile digest、expiry 和 256-bit nonce。客户端只看到 binding digest，不得到 nonce。
- Broker 拒绝同 tool call ID 的 scope/arguments/profile 重绑定；resolve 必须同时匹配
  authenticated principal、session 和 digest。decision、timeout 和 wait 都一次消费，
  因此 late callback 与 replay 失败。
- Scheduler dispatch 前必须调用 `consume_for_dispatch`；该操作兼容 Gateway 已经
  `resolve`/`wait` 的远端确认和 TUI 本地确认，但每个 binding 只允许一次 dispatch。
- Go Gateway 只从认证 middleware context 获取 principal；request body 的 principal 不
  具有权威性。confirm、task approve/reject 缺少认证或 binding digest 时返回 401/400。
- Plan Approval 已有 signed durable receipt 保持不变。普通 tool approval restart 后失效；
  destructive operation/ChangeSet 使用 durable one-shot consume ledger。

## 5.10 Structured Context 与 Compaction Window

- `ContextLayer` 明确区分 immutable rules、durable facts、conversation、conversation
  summary 和 ephemeral observation。Task/workspace/approval/plan/ChangeSet/verification
  事实每轮从权威对象重建；模型摘要不具有状态或授权效力。
- 仓库文件正文使用 `<untrusted_repository_content>` 边界作为 user observation 注入，
  不能伪装为 system rule。
- Compactor 不截断或摘要 system、durable fact、tool call 和 tool result，也不把这些内容
  发送给压缩模型。无法证明配对的 tool item 按保留处理。
- `context.compacted` 事件只记录 input window SHA-256、result SHA-256、level、token 计数
  和替换数量，不记录可能含 secret 的正文；该事件与 turn terminal 共用单调 sequence。

## 6. 错误码定义

## 5.9 Coding Runtime 对齐实现边界

Coding runtime 的新增能力采用独立适配层：Worktree/ChangeSet 负责隔离与审批绑定，ExecutionBackend 负责统一执行边界，LanguageRegistry/IndexStore 负责结构化代码检索，VerificationPipeline 负责分层验证。Legacy Parser、`test_run`、Verify-Fix 和既有权限/沙箱入口继续保留。

### 5.9.2 Implementation Planning（M4 Batch 0）

`coding/planning` 提供不可变的 `ImplementationPlan`、影响/验证/风险/诊断/证据模型以及只读 `PlanningService` 契约。计划只能基于 `CodeQueryService`、Resolution graph、`IndexStore` 和受信任仓库元数据生成；每个识别出的文件或 Symbol 都绑定可复核证据。服务不持有 `ToolScheduler`、ExecutionBackend、Worktree、ChangeSet 或 ApprovalBroker，因此不能写入用户仓库、执行命令、创建 ChangeSet 或自行审批。

相同仓库元数据、查询结果、目标和配置会产生相同的规范化目标、影响范围、验证要求、风险和 content hash；时间戳不参与 hash。HEAD、repository generation、文件 hash 或证据 Symbol 漂移会使计划变为 stale，stale 计划不得进入执行阶段。动态/歧义/缺失的解析结果必须降级为 diagnostic，且不得编造 stable symbol ID。验证命令只能来自受信任仓库元数据或既有验证管线，不能由目标文本或未来 LLM 候选直接指定。

### 5.9.3 Planned Workspace Mutation（M4 Batch 3.0）

计划执行只允许通过 `ApprovalRuntime → PlannedExecutionGuard →
WorkspaceMutationEngine`。输入是服务端重新计算摘要的结构化 Edit Bundle，操作限制为
create、update、delete、rename；执行根固定为 active `TaskWorkspace.worktree_path`，
不得接受调用方提供的任意 filesystem root。每次 edit 都重新验证 active Context、Lease、
Boot、Task/Workspace、Plan binding 和 mutation fence。

文件路径由固定 Workspace root directory FD 逐级通过 `O_DIRECTORY | O_NOFOLLOW`
解析；stat、hash、open、link、unlink 和 rename 都相对于已打开的父目录 FD。Create 和
rename 使用 true no-replace 原语，竞争者文件不得被覆盖；平台无法提供等价保证时
fail closed。Create 禁止 execute bit，update 不得新增 execute bit 或危险 mode。

同一原语通过 `SafeWorkspaceFS` 提升为 Coding 通用文件 capability。read/write/patch/
multi-edit/copy/move/search/symbol handler 只能取得 active TaskWorkspace 相对路径；拒绝
`.git/.agents/.codex/.khaos`、symlink、hardlink 和非 regular file。用户配置使用 0600
同目录临时文件、file fsync、dirfd rename 与 directory fsync，禁止直接 `write_text`。

每个 Edit 按 journaled、mutation-started、filesystem-applied、directory-synced、applied
推进。修改前必须先把路径、before/after hash、mode、状态及恢复 artifact 标识写入
SQLite Journal；源码正文
只允许进入 Worktree 外的私有 `.khaos-recovery/<execution_run_id>/`，不得进入数据库、
Audit 或普通日志。即使 mutation syscall 后的目录 fsync 或 event 写入失败，也按 Journal
和文件实际 hash 识别并恢复当前 Edit。全部 edit 完成后还须在同一 fence 内复核 HEAD、
generation、Git index、tracked/untracked 集合和 Worktree admin identity；随后进入 sealing，
只有 recovery artifact 删除并同步父目录、seal marker 持久化后才能进入 mutated。

失败时按 Journal 逆序 rollback；rollback 只处理匹配已知 before/after 状态的文件，第三方
内容出现时不得覆盖或删除。无法证明恢复完整时按 run/reason 独立 poison Workspace，
后续 acquire/index/cleanup/planned mutation 全部 fail closed，直到受控 recovery 完成。
启动恢复持有同一 Workspace mutation fence，并且只能清除当前 run 拥有的 quarantine。

### 5.9.4 Mutation terminal attestation（M4 Batch 3.0.2）

Rollback 的终态采用 `rolling-back → rollback-sealing → rolled-back/cancelled`：反向恢复
完成后先清理并同步 recovery artifact，再持久化 rollback seal marker，最后提交终态。
cleanup、Audit 或数据库提交失败时保持 recovery evidence，run 与 Workspace 均进入
poisoned；不存在“rolled-back 但 artifact 或 quarantine 仍存在”的可接受状态。

Mutation 进入 sealing 前必须持久化 `FinalMutationAttestation`。证明绑定 run、bundle、
Context、Lease、plan binding、HEAD、repository generation、Git index、Worktree admin
identity，以及每个声明路径的最终 exists/hash/mode；canonical digest 不包含源码、绝对路径
或调用方数据。允许路径不会被最终扫描豁免，create/update/delete/rename 的真实最终状态必须
逐项与 Journal 一致。启动恢复发现 sealing run 时重新验证 attestation digest、完整 Journal、
所有声明路径和仓库状态，任何缺失或漂移均保持 quarantine，不能猜测 MUTATED。

Workspace 必须由服务端配置仓库外 `recovery_root`。首次 backup 前验证其不位于 Worktree、
repository 或 Git administrative root 内；`RecoveryDirectory` 固定打开 container/run dirfd，
使用 `O_DIRECTORY | O_NOFOLLOW` 完成 artifact create/read/unlink，只处理 Journal 登记的随机
服务端名称，未知文件不会递归删除。Backup 顺序固定为 write/flush/fsync file、fsync recovery
directory、提交 Journal、执行 Workspace mutation；Journal 提交失败时不会修改 Workspace。

### 5.9.5 Terminal seal commit（M4 Batch 3.0.3）

Mutation 与 rollback/cancelled 终态都采用仓库外 durable tombstone。Source recovery artifact
清理并同步后写入只包含 run、bundle、attestation、Journal 和 container identity digest 的
`seal-<server-id>.json`，同步 tombstone 与 container 后，由一个 `BEGIN IMMEDIATE` 同时提交
seal marker、terminal status、completed time 与 Audit。事务成功后 tombstone 才进入幂等 GC；
因此 run directory 已删除时，启动恢复仍可用 attestation、canonical Journal 和 tombstone 完成
确定性终态提交，缺失或损坏证明则保持 quarantine。

Rollback 在进入 rollback-sealing 前持久化 `RollbackFinalAttestation`，逐项证明 create 路径
不存在、update/delete 已恢复 before hash/mode、rename source 已恢复且 destination 不存在，
并绑定完整 mode-aware Workspace state、HEAD、generation、index、admin、Context、Lease、
binding、reason 与 canonical Journal digest。恢复阶段必须重新验证该证明。

所有 persisted recovery input 在任何 dirfd 或 artifact open 前重新解析为服务端安全类型：
Workspace path 必须是 NFC 仓库相对 POSIX 路径；run、artifact、tombstone 必须匹配固定单组件
格式；operation 与连续 ordinal 也参与 canonical Journal digest。Git/Workspace state 对普通文件、
symlink 分别记录 content hash、mode、type、symlink target digest 和无路径 inode identity digest，
因此未声明 chmod、type 或 identity 漂移均会阻止 sealing。

### 5.9.6 Durable initial baseline（M4 Batch 3.0.4）

Execution run 在任何 recovery artifact、Journal 或 mutation 前读取两次稳定的 mode-aware
Workspace snapshot，并原子持久化 `InitialWorkspaceAttestation` 与 run baseline marker。
Live rollback 与 startup rollback 都只能使用该 persisted baseline；旧 incomplete run 缩缺证明时
fail closed。恢复 Journal 先转换为不可变 `ValidatedRecoveryJournal`，严格校验 NFC 原文、hash、
mode、operation 字段组合、durable phase、连续 ordinal、正 edit_count 和 path 冲突，后续路径、
artifact、attestation 与 tombstone 只消费 validated event。

Recovery storage 由长生命周期 `RecoveryRootCapability` 从可信系统根逐级 `O_NOFOLLOW` 打开并
固定 dev/inode/owner/mode，run 只复制 capability FD，不再重新解析完整 ancestor 路径。恢复终态
通过单个事务提交 seal、terminal status、Audit，并只删除对应 `run:<execution_run_id>` poison；
Runtime 启动会对已有合法 terminal seal 的遗留 run-owned poison 做幂等 reconciliation。

### 5.9.7 Baseline-bound rollback（M4 Batch 3.0.5）

`InitialWorkspaceAttestation` 是 rollback 唯一的 before-state 权威来源，并额外绑定服务端
批准的 canonical Edit after-state。恢复 Journal 的 source/destination、operation、before
hash/mode/type、after hash/mode 和 artifact 必须逐项与 initial proof 一致；0644/0600、
0755/0777 等 mode 漂移均 fail closed。Update/delete/rename 的全部 artifact 会在任何 Workspace
回写前通过固定 RecoveryDirectory capability 安全读取，并直接与 baseline content hash 比较；
artifact 不提供恢复权限，恢复 mode 只能取自 baseline。

合法 baseline 后、首条 Journal 前的启动崩溃采用独立 zero-journal 分支。只有 durable Journal
progress 为零、run recovery 目录缺失或为空、无 tombstone、两次 Workspace/Git snapshot 稳定且
与 baseline 的 HEAD、generation、index、admin、声明及未声明文件状态完全相同时，才能以
`no-mutation-crash` 安全终止；任何 artifact 或状态漂移都保持 quarantine。

所有 startup recovery 终态使用同一事务族：在 `BEGIN IMMEDIATE` 内重新验证 canonical
attestation、CAS Run 状态、提交 seal/no-mutation marker、terminal status、completed/failure、
Audit，并只删除 `workspace_id + run:<execution_run_id>` poison。内存 fence 只在数据库提交后
同步。Edit Journal 通过 CAS 状态机推进：`journaled → mutation-started → filesystem-applied →
directory-synced → applied`；已发生文件系统变化的 Edit 回滚必须经 `rollback-started →
rolled-back`，未变化的 journaled/mutation-started Edit 经验证后可直接 rolled-back。每次推进
同时校验 Run phase、phase version 和 immutable after-state，拒绝跳跃、倒退和并发重复获胜。

### 5.9.8 Restart-idempotent rollback ownership（M4 Batch 3.0.6）

Run rollback 通过 `begin_or_resume_rollback()` 单事务开始或续跑。validating、mutating、
poisoned 原子进入 rolling-back 并只写一次 rollback-started Audit；rolling-back 保持原状态并
保留首次 failure reason；rollback-sealing 交给 seal recovery；rolled-back/cancelled 直接返回
终态；mutated/failed 拒绝回滚。不同重试原因不能覆盖首次原因。

Edit recovery 按 durable phase 分流：journaled 与未发生 syscall 的 mutation-started 只验证
initial baseline 后直接 rolled-back；filesystem-applied/directory-synced/applied 先 CAS 到
rollback-started；rollback-started 根据 ownership proof 安全续跑；rolled-back 只复核 baseline
和 rollback identity，不再次执行文件操作。same-phase 重试必须同时匹配 error code、after-state
和 identity，成功时 phase version 与 Audit 均不增加。

每个 mutation event 持久化 applied object、source parent、rename destination、rollback object
identity digest 和 identity version。Digest 只包含 st_dev/st_ino、file type、父目录 identity、
run/edit/operation/role，不包含绝对路径。Mutation syscall 后立即通过固定 dirfd lstat，并在目录
fsync 前原子提交 identity + filesystem-applied；rollback syscall 后同样先提交 rollback identity，
再进入 rolled-back。syscall 已发生但 identity 尚未提交的窗口不得按 hash/mode 猜测 ownership，
必须以 identity-evidence-missing 保持 quarantine。

FinalMutationAttestation 对 create/update/rename 绑定 persisted applied identity，对 delete 绑定
不存在状态和受控 parent identity。相同内容、相同 mode 但不同 inode 的第三方 create/update/
rename replacement 或 delete re-creation 均不能被 rollback 删除、覆盖或移动；恢复证据保留，
Run 和 Workspace fail closed。

### 5.9.9 Durable rollback directory sync（M4 Batch 3.0.7）

发生实际 filesystem rollback 的 Edit 使用独立 durability 链：`rollback-started →
rollback-filesystem-applied → rollback-directory-synced → rolled-back`。syscall 成功后通过固定
dirfd 重新读取 rollback object 与父目录 identity，并在任何目录 fsync 前原子持久化 object、
source parent、rename destination parent、sync mask、phase version 和 Audit。所有所需父目录
完成 fsync 后，第二个事务写入 directory sync digest、synced timestamp 和 identity version；
只有该证明完整时才能进入 rolled-back。未发生 syscall 的 journaled/mutation-started Edit 仍可在
与 InitialWorkspaceAttestation 的 hash、mode、type、inode 全部一致时直接 rolled-back。

create rollback 删除目标并同步 source parent；update rollback 原子恢复 baseline 文件并同步 source
parent；delete rollback 重建 baseline 文件并同步 source parent；rename rollback 恢复 source、移除
destination，同目录只同步一个唯一 parent，跨目录同步 rollback source 与 destination 两个 parent。
sync mask 和 parent identity digest 由服务端从固定目录句柄生成，不接受 Bundle 输入。

启动恢复遇到 rollback-filesystem-applied 时只验证 persisted ownership 并幂等重试目录 fsync，不重放
delete、replace、create 或 rename syscall；rollback-directory-synced 只复核 object/parent identity 后
推进 rolled-back。Batch 3.0.6 旧 incomplete row 若仅有 rollback identity，则先验证现存对象，补建
parent/sync requirement 后重新 fsync；无法证明 ownership 时保持 quarantine。旧 terminal row 不迁移。
任一 fsync、identity revalidation 或 phase CAS 失败都保留 recovery evidence 和当前 durable phase，
Workspace 继续 poisoned。

本阶段不执行 verification、Tool、terminal/test_run，不生成或应用正式 ChangeSet，也不做
Git commit/push。现有 M2 file tools、WorkspaceManager ChangeSet 和 Git 工具继续遵循各自
权限、沙箱与审批规则，不能被 planned execution capability 替代。

### 5.9.1 LanguageRegistry 与解析契约

`coding/intelligence` 内的解析结果统一使用 `ParseResult`。内部行号和列号均为
0-based；列号表示 Unicode code point 数，不是 UTF-8 byte column，也不是 LSP
使用的 UTF-16 code unit。`byte_start` 和 `byte_end` 是原始内容 UTF-8 bytes 的
半开区间。旧接口的 `line` 字段继续通过兼容层输出 1-based 值。

语言只由锁定的扩展名注册表解析，不接受用户提供的任意语言字符串。Tree-sitter
及 grammar 都是可选的锁定依赖或预构建包：运行时不得联网安装、不得从用户项目
目录搜索 grammar、不得动态加载项目内 native library。Python 的降级顺序为
Tree-sitter、stdlib AST、Legacy Regex；JavaScript、TypeScript、Go、Rust 为
Tree-sitter、Legacy Regex。任何可恢复失败都写入 structured diagnostic，不能
中止仓库级索引。

可选依赖不可用时必须返回降级诊断，不得运行时联网安装；可写 Task 默认只使用独立 Worktree，主工作树回写必须经过 ChangeSet 审批。

### 5.9.10 可信验证执行（M4 Batch 3.1）

规划验证与 Agent `terminal` / `test_run` 是两条独立授权链。可信验证命令只能由
`VerificationCatalog` 根据已批准的 `VerificationRequirement` 生成，并绑定 Plan、
Approval、Execution Run、最终 mutation attestation、catalog fingerprint、工具链和
Sandbox profile。调用方不能提供或覆盖 argv、cwd、env、timeout、镜像或 expected exit
code；生产执行拒绝 shell launcher、Workspace executable、动态安装和联网。

项目代码只在一次性 Verification Workspace 中运行。该副本位于主仓库、TaskWorkspace、
recovery root 和审批数据库之外，不复制 `.git`、symlink、hardlink、恢复证据或凭据；复制
前后以 hash/mode/type manifest 核对。生产 `SandboxBackend` 必须固定镜像 digest、绝对
executable 和工具链 manifest，使用只读 rootfs、非 root、drop capabilities、
no-new-privileges、network=none、资源上限与私有 tmpfs。强隔离后端不可用时 fail closed，
不得回退到宿主进程。

验证使用同一个逻辑 `PlanExecutionRun` 的 verification phase continuation lease。运行前和
每一步前重新验证 persisted Plan、MUTATED 状态、FinalMutationAttestation、bundle/binding、
HEAD/generation/index/admin、catalog/config/toolchain/profile digest。验证状态和 Execution
Run 终态使用同一 SQLite `BEGIN IMMEDIATE` CAS 事务提交；崩溃中的 preparing/running
步骤只能恢复为 errored，不能推断为通过。stdout/stderr 有硬上限、脱敏和私有 artifact
边界，数据库与 Audit 只保存 digest、长度、截断标记和脱敏摘要。

Batch 3.1.6 将审批时的验证供应链固化为不可变
`ApprovedVerificationPlanSnapshot`。加载时必须从数据库全部字段重算 canonical
digest，并与 ApprovalRequest 绑定值比较；执行时还要精确比较 image、
toolchain、binary、version output、parsed version、profile 和 policy fingerprint。
快照只允许 immutable insert，相同 ID 的不同内容必须冲突，已被 ApprovalRequest
引用的快照不得更新或删除。

生产成功终态必须经过 `VerificationCleanupProof`：Disposable 必须已 CLEANED
且具有持久化 `cleaned_at`，Sandbox 必须有真实 absence proof，Artifact 必须已
SEALED 且实物 hash/size/type 匹配，canonical Workspace 必须仍等于最终 mutation
attestation。`FINALIZING` 启动恢复重做全部证明检查，仅在证据完整时以
单个 SQLite 事务完成 PASSED/VERIFIED。

Artifact 与 Verification Workspace 存储不再接受调用方 Path 或 factory。
`ApprovalRuntime` 在当前 boot 内签发 opaque storage capability ID，内部 registry
通过逐级 `O_DIRECTORY | O_NOFOLLOW` 打开并固定 root FD，绑定
dev/inode/uid/gid/mode，且拒绝与 repository、main worktree、TaskWorkspace、
recovery root、SQLite root 及彼此重叠。伪造、跨 Runtime、旧 boot 或 root
identity 漂移的 capability 一律在创建进程前 fail closed。

Batch 3.1.6.1 将成功终态收敛为唯一入口：只有
`VerificationExecutionStore.finalize_success()` 可在同一 `BEGIN IMMEDIATE`
事务中写入 Verification Run `PASSED` 和 Execution Run `VERIFIED`。通用
transition、旧 `finish_step_and_run()` 和直接 SQL 均不得产生成功；SQLite
trigger 通过连接内部的短生命 finalization guard 与 run-scoped CleanupProof
双重校验。第二个成功 CAS、proof 校验或 Audit 失败时，整个事务
回滚，不允许两个成功状态分裂。

`VerificationCleanupProof` 以 `verification_run_id` 唯一且写入后不可变：
同一 run 的相同 canonical 内容是幂等写，不同内容是 immutable
conflict；UPDATE/DELETE 由 trigger 拒绝。旧库如存在同一 run 多条 proof，
migration 必须 fail closed，不得选择“最新”记录。

Artifact 最终化与 `FINALIZING` 恢复共用同一安全文件证明实现：
从已固定 storage root FD 使用 `O_RDONLY | O_NOFOLLOW | O_NONBLOCK |
O_CLOEXEC` 相对打开，然后以 `fstat()` 精确验证 regular file、
`nlink == 1`、owner、`0o600`、dev/inode、size 和 SHA-256。读取前后通过
parent dirfd 重新 stat basename，目录项替换、symlink、hardlink、FIFO、
socket、directory 或 device 均非阻塞地 fail closed。Artifact seal digest 同时
绑定内容与这些 identity 字段。

#### 5.9.10.1 Verification Write Authority 与威胁边界

Batch 3.1.6.2 不再使用 connection-local SQLite UDF 识别可信写入者。
UDF、collation、extension 和 connection object identity 都可被同权限调用者
伪造，因此 trigger 只保留状态机一致性、CleanupProof 存在性和不可变
证据约束，不再承担连接鉴权。

生产 `ApprovalRuntime` 在每个 boot 中只签发一个
`VerificationWriteAuthority`。Authority 绑定 runtime ID、boot ID、SQLite DB、
WAL、SHM 和 parent directory 的 dev/inode/uid/gid/mode，逐级以
`O_DIRECTORY | O_NOFOLLOW` 打开父目录。在 Runtime 活跃期间，DB/WAL/SHM
路径权限降为 `0o400`，只有 Authority 启动前已打开的 SQLite writer FD
可继续提交；所有可信读写前后重新验证 storage identity 和 schema/trigger
digest。替换 DB/WAL/SHM、parent rename/symlink swap 或 schema drift 都 fail
closed。非 Authority 读取只能使用固定查询的 `VerificationReadHandle`，
内部连接使用 `mode=ro` 和 `query_only`，不暴露 raw connection/path/factory。

CleanupProof 的首次写入由 Authority 记录 boot-scoped 绑定；
`finalize_success()` 必须同时校验持久 proof 和 Authority 绑定。成功事务
另写入 immutable `verification_success_evidence`，绑定 run、execution、proof、
runtime、boot 和 authority instance。可信读取不仅检查普通
`passed/verified` 缓存，还必须得到当前 Authority 内存证据确认。新
boot 对旧成功状态不猜测 PASS：缺少当前 boot 权威锚点的历史
成功会转为 `ERRORED/VERIFICATION_ERROR` 并写入审计，需要重新
验证而不是自动恢复为成功。

Authority 子进程的 proof/success ledger 跨启动保留，但所有记录以 boot ID 隔离；旧
boot 记录只用于审计，不能满足当前 boot 的 require。boot start/stop、accepted 和
rejected RPC 写入 previous-hash 串联的 canonical event chain，启动签发 Authority 前
必须验证完整 SHA-256 链。proof/success 状态与 accepted event 在同一事务提交。

当前 Khaos Desktop 为单 OS 用户、单信任 Runtime 进程模型，本边界防御：
仅能通过公开 API 调用的插件、只读组件、Sandbox/项目代码、
不持有 Authority FD 的独立进程，以及在 Runtime 活跃期间对 DB
路径的普通写尝试。不防御：在 Khaos 可信 Runtime 进程内执行的
恶意 Python、能调试/修改该进程的同 UID 主体，或 root/操作系统管理员。
同 UID 文件所有者理论上可重新 chmod 文件；这属于 OS authority breach，
不声称由 SQLite 阻止。如未来要覆盖该威胁，必须将 writer 迁移到
不同 UID/独立 broker 进程，使用 peer credential 验证的 Unix socket 或继承
匿名 pipe；不得降级回 UDF、同进程 secret 或可预测 bearer token。
